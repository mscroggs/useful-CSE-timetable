Implementing a SYCL Backend for Kokkos
Daniel Arndt
Monday, February 27 1:50 PM–2:05 PM
This is the 1st talk in SYCL's Impact on Algorithms, Data Structures and Implementations - Part II of II (1:50 PM–3:30 PM)
G104

Aurora, to be deployed at Argonne National Laboratory, is expected to be the next exascale computer in the United States. Just as with Frontier, the first exascale computer at Oak Ridge National Laboratory, Aurora's main performance will come from GPUs by a new vendor. Kokkos' (main) choice for supporting these GPUs developed by Intel is leveraging DPC++, Intel's implementation of the SYCL standard. In this talk, I discuss my experience while developing a SYCL backend for Kokkos and issues applications using this backend encounter. In particular, I will compare features that work better or worse in Kokkos than with the HIP or CUDA backend.

Flavours of GPU Kernels in ExaHyPE
Tobias Weinzierl
Monday, February 27 2:10 PM–2:25 PM
This is the 2nd talk in SYCL's Impact on Algorithms, Data Structures and Implementations - Part II of II (1:50 PM–3:30 PM)
G104

ExaHyPE is an engine to solve hyperbolic PDEs in first order formulation over dynamically adaptive meshes. The code supports classic non-overlapping domain decomposition supplemented by tasks. At runtime, ExaHyPE assembles batches of tasks and deploys them to GPUs. For various numerical schemes, we offer various generic, i.e. PDE-independent GPU kernels. By default, they rely on OpenMP GPU offloading. This talk reflects on our recent endeavour how to translate these OpenMP kernels into SYCL. Some concepts from the OpenMP world translate one-to-one, while others cannot directly be mapped onto SYCL. In return, SYCL streamlines the realisation of certain flavours of GPU kernels which are technically challenging in OpenMP. The two paradigms require a rethink of the kernel realisations.

Exploration of Performance-Portability in the ExCALIBUR Fusion Use Case
Will R. Saunders
Monday, February 27 2:30 PM–2:45 PM
This is the 3rd talk in SYCL's Impact on Algorithms, Data Structures and Implementations - Part II of II (1:50 PM–3:30 PM)
G104

Project NEPTUNE approaches one of the grand challenge problems in fusion - modelling plasma in the edge region of a tokamak. In this edge region, hot plasma interfaces with the reactor wall and cold neutral gas resulting in a computationally expensive multi-scale problem that is considered an exascale challenge. For the fusion community it is highly desirable to implement the computational methods that simulate this multi-scale model in a performance-portable manner. The NESO (Neptune Exploratory SOftware) implementation is an intermediate project to evaluate the suitability and performance of proposed techniques and algorithms for NEPTUNE. As part of NESO we are evaluating SYCL as a performance-portable language to target multiple hardware architectures. In this talk we present how SYCL is used in the NESO project and our proposed approach for sustaining performance across different hardware architectures in the fusion use case.
